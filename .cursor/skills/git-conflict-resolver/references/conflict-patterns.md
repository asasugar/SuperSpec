# Git冲突模式识别

## 常见冲突模式

### 1. 导入语句冲突
**模式特征:**
- 两个分支都添加了新的导入
- 导入顺序不同
- 导入位置冲突

**示例:**
```
<<<<<<< HEAD
import { ComponentA } from './components/A';
import { ComponentB } from './components/B';
=======
import { ComponentC } from './components/C';
import { ComponentA } from './components/A';
>>>>>>> feature-branch
```

**解决策略:**
- 合并所有导入
- 按字母顺序排序
- 移除重复导入

### 2. 函数参数冲突
**模式特征:**
- 函数签名被修改
- 参数顺序不同
- 参数类型不同

**示例:**
```
<<<<<<< HEAD
function processData(data: string, options: Options) {
=======
function processData(data: string, callback: Function) {
>>>>>>> feature-branch
```

**解决策略:**
- 合并所有参数
- 保持向后兼容
- 更新函数调用

### 3. 配置冲突
**模式特征:**
- 配置文件被修改
- 添加了新的配置项
- 配置值不同

**示例:**
```
<<<<<<< HEAD
{
  "timeout": 5000,
  "retries": 3
=======
{
  "timeout": 3000,
  "cache": true
>>>>>>> feature-branch
}
```

**解决策略:**
- 合并所有配置项
- 保留双方有价值的配置
- 统一配置值

### 4. 样式冲突
**模式特征:**
- CSS/样式文件冲突
- 类名冲突
- 样式规则冲突

**示例:**
```
<<<<<<< HEAD
.button {
  color: blue;
  padding: 10px;
=======
.button {
  color: red;
  margin: 5px;
>>>>>>> feature-branch
}
```

**解决策略:**
- 合并所有样式规则
- 保留双方样式
- 统一设计规范

### 5. 测试冲突
**模式特征:**
- 测试用例被修改
- 添加了新的测试
- 测试逻辑不同

**解决策略:**
- 合并所有测试用例
- 确保测试覆盖完整
- 更新测试数据

## 自动识别模式

### 模式匹配规则

1. **导入冲突**: 检测 `import` 关键字
2. **函数冲突**: 检测函数定义模式
3. **配置冲突**: 检测 JSON/YAML 结构
4. **样式冲突**: 检测 CSS 选择器
5. **测试冲突**: 检测测试框架关键字

### 智能合并建议

- **导入**: 自动去重和排序
- **函数**: 合并参数，保持兼容性
- **配置**: 深度合并对象
- **样式**: 合并CSS规则
- **测试**: 保留所有测试用例

## 处理优先级

1. **高优先级**: 核心业务逻辑冲突
2. **中优先级**: 配置和工具函数冲突
3. **低优先级**: 样式和测试冲突

## 批量处理策略

对于相同类型的冲突:
1. 识别冲突模式
2. 应用统一解决策略
3. 批量处理相似文件
4. 验证处理结果
